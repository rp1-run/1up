## Methodology

- **Search tests (4):** Trace multi-file flows across the emdash monorepo. Tests conceptual exploration where the agent doesn't know file paths upfront.
- **Impact tests (3):** Identify blast radius of changing a specific file/symbol. Tests dependency tracing through call sites and type references.
- **1up agent:** Has access to `1up search`, `1up symbol`, `1up context`, `1up impact`, plus `grep` per the decision table.
- **Baseline agent:** Has access to `grep`, `rg`, `find`, `cat`, `Glob`, `Grep`, `Read` — standard CLI tools only.
- **Assertions:** File-matching (expected files in output), LLM rubric (accuracy + completeness scored by Haiku), efficiency reporting (time, cost, turns).
- **Fixture:** Each test gets a fresh copy of the emdash repo at a pinned commit with a pre-built 1up index.

## Notes

- Sonnet reports 2 turns because the Agent SDK counts the full agentic conversation as a single turn. Actual tool calls within each turn are 8-30+.
- FTS Impact is the one baseline win — the task gives a specific class name (`FTSManager`), so grep is competitive. Consistent with the decision table: "Symbol/keyword name -> grep."
- Single-run results. Sonnet shows lower variance than Haiku but results may differ across runs.
